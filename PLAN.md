## Фаза 0 — Walking skeleton (= MVP-0). Делать ПЕРВОЙ.

Цель: `python -m crosspost post --to telegram,vk` реально постит. Без очереди/веба/БД/браузера.

1. **Рецепт адаптера** (зафиксировать как процедуру — по нему клонируются все каналы):
   1. скопировать `adapters/api/telegram.py` → новый канал;
   2. скопировать `tests/adapters/api/test_telegram.py` → тест канала (мок площадки);
   3. поменять рендер (canonical → формат канала) и вызов отправки;
   4. вернуть `ChannelResult` с квитанцией; `store.mark_done(...)`;
   5. зелёный тест на моках — канал готов, реальные ключи не требуются.

2. **TG-адаптер** (фаза A, под TDD): сделать красный тест зелёным — рендер + `send_file` через Telethon + квитанция (`message_id`) + `mark_done`.

3. **ВК-адаптер** по рецепту: `wall.post` + загрузка фото; мок VK API; зелёный тест.
   > **Статус (MVP-0):** адаптер реализован, текстовый пост работает. Фото заблокировано:
   > community-токен отдаёт VKAPIError 27 на оба метода загрузки. Флаг
   > `VK_PHOTO_UPLOAD_ENABLED=false` в `runtime/.env` обходит ограничение.
   > **Развилка на будущее (выбрать перед MVP-1/MVP-2):**
   > - A. User-токен (`wall+photos+offline`) → фото остаётся в API-тире (предпочтительно).
   > - B. Playwright → фото ВКонтакте вместе с эпиком 5, не раньше (добавляет весь браузерный
   >   стек: profile/лок/verify-before-retry/breaker).
   > До выбора ветки ВК остаётся текстовым через API.

4. **Сборка CLI**: реализовать `build_adapter()` в `__main__.py` — собрать TG и ВК из `runtime/.env`. Поканальный вывод уже готов.

5. **Smoke**: заполнить `runtime/.env` (5 ключей из SETUP.md), `make smoke` → пост уходит в оба канала.

**Выход Фазы 0:** два канала постят из CLI, состояние в JSON, идемпотентность работает. Это MVP-0.

## Фаза A (детали TG/каркаса — влиты в Фазу 0 выше; раздел сохранён для истории)

## Фаза A — каркас + первый канал (= MVP-0)

Цель: загрузил контент → ушло в TG → вижу результат. Тончайшая сквозная вертикаль.

1. **Каркас (эпик 0).** Структура репозитория, `.env`, схема состояния задач, статусная модель. Контракт `async def publish(canonical) -> ChannelResult` как абстрактный класс/протокол. Две сущности состояния: Task и ChannelConnection.
2. **Контент минимум (эпик 1).** `CanonicalContent` с `type`, валидация для `post`, capability-матрица. Рендер — в файле TG-адаптера (без отдельного `Renderer` пока).
3. **TG-адаптер (эпик 2) под TDD.**
   - Сперва тест: мок Telethon → публикация возвращает квитанцию; повторный вызов с тем же `publication_id` не создаёт дубль (дедуп по внутреннему ключу, не по квитанции).
   - Затем реализация: `StringSession`, постинг от лица канала, текст+картинка.
4. **Тончайший триггер.** Пока без UI и очереди — CLI или один FastAPI-эндпоинт: принять контент → вызвать TG-адаптер синхронно → вернуть `ChannelResult`.

**Выход A:** один канал работает сквозняком, контракт проверен на реальном адаптере.

## Фаза B — оркестрация + остальной API-тир (= MVP-1)

Цель: 4 API-канала, асинхронные поканальные статусы, ретрай только упавшего.

5. **Оркестратор + очередь (эпик 3).** С появлением >1 воркера перейти со SQLite на Postgres (write-lock). Раскладка отправки на атомарные поканальные задачи; Redis + Celery/RQ; ретраи с backoff и потолком попыток; circuit breaker и проверка connection перед диспатчем; агрегация статусов; раздельные fast/slow дорожки (slow пока пустая); ретрай только упавшего канала.
6. **Остальные API-адаптеры (эпик 2)**, каждый под TDD по шаблону TG:
   - VK → Telegraph → YouTube.
7. **Медиа-цикл (эпик 7).** Локальный temp; чистка по финалу всех каналов.
8. **Веб-интерфейс (эпик 4).** Форма + чекбоксы + «Отправить»; асинхронная панель статусов «загораются по одному»; кнопка «Повторить только его». Предварительный статус канала пока тривиален (для API он почти всегда «жив»).

**Выход B:** полноценный API-тир, частичный успех и точечный ретрай работают, экран не блокируется.

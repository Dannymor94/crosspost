# PRD / Бэклог — Кросс-постинг

**Источник:** PROJECT_GUIDE.md v1.1
**Версия бэклога:** 1.1
**Принцип порядка:** сначала API-ядро (ни от чего не зависит), браузерный тир и его инфраструктура — позже.

> Изменения v1.1: идемпотентность по внутреннему ключу; разведены ChannelConnection и Task; добавлены capability-матрица, circuit breaker, потолок ретраев, TTL медиа; контракт async; БД по конкурентности записи.

## Вертикальные срезы (что считаем «работает»)

- **MVP-0 — тонкая вертикаль:** загрузил контент → ушло в один канал (TG) → видишь результат. Эпики 0, 1 (минимум), задача TG из 2, минимум 3 и 4.
- **MVP-1 — весь API-тир:** 4 API-канала, поканальные статусы, точечный ретрай. + остаток 2, полные 3 и 4, эпик 7.
- **MVP-2 — браузерный тир:** один браузерный канал сквозняком, затем остальные. Эпики 5, 6.
- **v1.0 — полный:** планировщик + мультиюзер. Эпики 8, 9.

Граница API↔браузер = граница между MVP-1 и MVP-2. До неё инфраструктура (vault, прокси, релогин, health-check, breaker) не нужна.

## Отложено после текущего этапа (НЕ реализовывать сейчас)

Follow-ups к уже начатому браузерному тиру (Яндекс сквозняком работает: login → storageState → publish → SUBMITTED). Взять после текущего этапа.

- [ ] **1. ВК → браузерный тир по образцу Яндекса.** API уперся в блокировки платформы (см. Эпик 2, ветка B). Инфраструктура браузерного тира уже готова: `open_page`/`login_context`/`storage_state_path`/лок/verify-before-retry. `VKBrowserAdapter` есть — довести по тому же паттерну, что и Яндекс (ручной `login --channel vk` → storageState → publish с verify-before-retry). Отдельная задача, не блокирует текущий этап.
- [ ] **2. Яндекс: реальная ссылка на публикацию вместо заглушки.** `_wait_for_card` сейчас при неудаче извлечения id возвращает `external_id="submitted"` (заглушка). Довести до реального URL/id опубликованной карточки — квитанция нужна для верификации/правки/удаления. Модерация до 7 дней осложняет: ссылка может появиться не сразу. Возможно, разнести на «отправлено» (SUBMITTED сейчас) и «подтверждено ссылкой» (позже, post-модерация — пересекается с health-check эпика 6).
- [ ] **3. ВК-фото/текст через user-токен как альтернатива браузеру.** Если решится official user-token (Standalone-app, `wall+photos+offline` — Эпик 2, ветка A) — ВК-фото остаётся в API-тире (чище и стабильнее браузера). Развилка A vs B: выбрать по факту доступности токена. Не делать оба параллельно.

## Эпик 0 — Каркас

- [ ] Репозиторий, структура пакетов, конфиг через `runtime/.env`, секреты вне кода
- [ ] **Две сущности состояния:**
  - [ ] `Task`: `new → queued → attempting → done | failed(reason) | held`
  - [ ] `ChannelConnection (user, channel)`: `live | needs_relogin | banned`
- [ ] Внутренний `publication_id` (генерится до отправки) как ключ идемпотентности
- [ ] **Контракт адаптера** `async def publish(canonical) -> ChannelResult` — единый для всех 8 каналов
- [ ] `ChannelResult`: `{status, external_id?, error?, channel}` — `external_id` = квитанция, не ключ дедупа

**Приёмка:** новый канал добавляется реализацией одного async-интерфейса, без правок оркестратора. Состояние подключения и состояние задачи — раздельные сущности.

## Эпик 1 — Контент-слой

- [ ] `CanonicalContent` с полем `type`: `post | article | reel | story`
- [ ] Валидация по типу (пост — текст+изображение; статья — заголовок+тело+обложка; reel/story — текст+видео)
- [ ] **Capability-матрица** `канал → {поддерживаемые type}`
- [ ] Рендер под канал (живёт в файле адаптера; интерфейс `Renderer` выделяется позже при дублировании)

**Приёмка:** один `CanonicalContent` даёт разный payload на разные каналы; матрица отсекает неподдерживаемые комбинации (reel→Telegraph) ещё до очереди.

## Эпик 2 — API-ядро

> Stateless, общий IP, релогин почти никогда, площадка возвращает квитанцию.

- [ ] **[СТАРТ] TG-адаптер (Telethon)** — постинг от лица канала, `StringSession`
  - Приёмка: пост уходит; вернулась квитанция; повторный вызов с тем же `publication_id` не создаёт дубль (дедуп по внутреннему ключу, не по `external_id`)
- [ ] VK-адаптер — `wall.post` + загрузка фото через wall upload server
  > **⛔ ЗАБЛОКИРОВАНО платформой (проверено вживую, post-MVP-0).**
  > - Community-токен: фото — ошибка 27, текст — ошибка 214 (нет прав постить на стену).
  > - User-токен: официальный OAuth-поток id.vk.com / Web-приложение не отдаёт Implicit-токен.
  > - VK-мессенджер: публичного API для постинга не обнаружено.
  > Код `VKAdapter` готов и покрыт тестами — ждёт рабочий токен/доступ.
  > **MVP-0 закрывается на Telegram.** Возврат к ВК — отдельной задачей после MVP:
  > - **A. Standalone-приложение (API-тир):** зарегистрировать Standalone-app в VK, получить
  >   user-токен с `wall+photos+offline` через Device Code или PKCE — остаётся API-тиром.
  > - **B. Playwright (браузерный тир):** перенести ВК вместе с эпиком 5 (WhatsApp/Instagram/
  >   Дзен/Яндекс). Тянет profile/лок/verify-before-retry/health-check. Не раньше MVP-2.
- [ ] Telegraph-адаптер — публикация статьи (`type=article`)
- [ ] YouTube-адаптер — Data API, заливка видео (`type=reel`); учесть суточную квоту

**Приёмка эпика:** все 4 канала реализуют async-контракт; идемпотентность по внутреннему ключу; работают на одном IP.

## Эпик 3 — Оркестратор + очередь

- [ ] Приём отправки → раскладка на **атомарные поканальные** задачи
- [ ] Очередь (Redis + Celery/RQ)
- [ ] Ретраи с backoff **и потолком попыток**, dead-letter
- [ ] **Проверка перед диспатчем:** подключение `live`? breaker не открыт? — иначе `held`/`failed`, не отправляем
- [ ] **Circuit breaker** на `(user, channel)`: бан/лимит → пауза, задачи не диспатчатся
- [ ] Агрегация поканальных статусов в один результат отправки
- [ ] **Раздельные fast / slow дорожки** — браузерный канал не держит API-канал
- [ ] **Ретрай только упавшего канала** — без повторной отправки в успешные

**Приёмка:** частичный успех корректен; повтор шлёт только в провалившийся; забаненный канал не ретраится; мёртвое подключение не диспатчится.

## Эпик 4 — Веб-интерфейс

- [ ] Форма: контент (текст+картинка / текст+видео) + чекбоксы каналов + «Отправить»
- [ ] **Чекбоксы по capability-матрице** — недоступные для данного `type` каналы выключены
- [ ] **Предварительный статус подключения** — `needs_relogin`/`banned` канал серый/красный
- [ ] Асинхронная поканальная панель: статусы **загораются по одному**
- [ ] Кнопка «Повторить только его» у упавшего канала
- [ ] Просмотр ошибки канала

**Приёмка:** отправка не блокирует экран; нельзя выбрать неподдерживаемый или мёртвый канал; статусы приходят по мере готовности.

## Эпик 7 — Жизненный цикл медиа

> Раньше браузерного тира, т.к. нужен уже при мультиканальной отправке.

- [ ] Локальное temp-хранилище загруженного медиа
- [ ] **Чистка только когда ВСЕ выбранные каналы финально завершились** (done/failed после ретраев)
- [ ] **TTL** — медиа не удерживается бесконечно из-за висящего `needs_relogin`-канала
- [ ] Привязка чистки к статусу задачи отправки, а не к событию канала

**Приёмка:** при «ВК ✓, IG висит» файл не удаляется; удаляется после финала по всем каналам либо по TTL.

---

## Эпик 5 — Браузерный тир

> Хрупкий, дорогой. Сначала ОДИН канал сквозняком как доказательство, потом остальные.

- [ ] Инфра фермы: **персистентные изолированные профили** (`user-data-dir`), не инкогнито
- [ ] Воркер поднимает профиль по требованию → постит → гасит
- [ ] **Лок `(user, channel)`** — одна задача на сессию в момент времени
- [ ] **Verify-before-retry** — маркер `attempting` до клика; при повторе сперва чтение ленты, потом дожим
- [ ] **Детекция бан-сигнала** → переключение подключения в `banned` (вход в circuit breaker)
- [ ] Решить headless vs headful (Xvfb) для сервера — задача-исследование
- [ ] Адаптер-пилот (Дзен — наименее баноопасный) — реализует контракт
- [ ] Остальные 3 браузерных адаптера (каждый отдельный, падение одного не роняет другие)
  - [ ] WhatsApp (Status / группа)
  - [ ] Instagram («добавить рилс»)
  - [ ] Яндекс Бизнес («Публикации», модерация до 7 дней)

**Приёмка:** дубли отсутствуют даже при падении воркера после клика; редизайн одной площадки ломает только её адаптер; бан-сигнал останавливает ретраи.

---

## Эпик 6 — Vault, health-check, релогин

> Нужны как только появляется браузерный тир и/или мультиюзер.

- [ ] **Session vault**: шифрование `storageState` at rest
- [ ] **Ключ шифрования инжектится снаружи (env/keyring/KMS), НЕ под `runtime/`**
- [ ] Доступ к vault — только воркерам, не фронту; расшифровка коротко в памяти
- [ ] **Сырые пароли не хранятся** — нигде
- [ ] **Проактивный health-check сессий** по крону, до отправки
- [ ] Мёртвая сессия → подключение `needs_relogin`, задачи в него не ставятся
- [ ] Страница «Подключения»: статус по каналу + кнопка «Переподключить»
- [ ] **User-driven релогин**: QR/код проходит сам пользователь; авто-логин паролем не используется
- [ ] Уведомление при покраснении канала

**Приёмка:** мёртвая сессия не приводит к тихой потере поста; релогин выполняет пользователь; ключ vault не утекает вместе с копией `runtime/`.

## Эпик 8 — Планировщик

- [ ] `отложка` — хранит запланированные задачи, отдаёт готовые по наступлении времени
- [ ] `пейсинг` — впускает в очередь с интервалом, не залпом (разнос нагрузки + анти-бот)
- [ ] Явная цепочка `отложка → пейсинг → очередь` (не два независимых скрипта)

**Приёмка:** отложенный пост уходит в заданное время; совпавшие отправки не идут залпом.

---

## Эпик 9 — Мультиюзерность (3–5 друзей)

- [ ] Модель пользователя; токены/сессии на пользователя
- [ ] API-каналы: хранение токена/StringSession на юзера (тривиально)
- [ ] Браузерные каналы: профиль + поле **липкого прокси на аккаунт** в профиле пользователя
- [ ] Изоляция данных пользователей в vault и состоянии задач

**Приёмка:** API-каналы масштабируются добавлением строк токенов; браузерные каналы каждого юзера ходят со своего стабильного IP.

### 9-A: Мультиаккаунт в браузерных каналах (детализация, не раньше эпика 9)

Проблема: сейчас `org_id` один на всё приложение (из `.env`). Нужна модель `(channel, profile, org_id)`.

Три сценария, которые нужно закрыть:

| Сценарий | Профиль | org_id |
|---|---|---|
| Разные логины Яндекса | `browser_profiles/<user>/<channel>/` | свой на юзера |
| Одна учётка, разные орг-ии | общий профиль | разный, на публикацию |
| Текущий MVP (один аккаунт) | `browser_profiles/<channel>/` | из `.env` (достаточно) |

Изменения при реализации:
- `build_adapter()` / CLI: принимать `--profile <name>` и `--org-id <id>` (переопределяют `.env`).
- Путь профиля: `browser_profiles/<profile>/<channel>/` вместо `browser_profiles/<channel>/`.
- `org_id` задаётся на публикацию/профиль, не глобально.
- Vault и `ChannelConnection` хранят ключ `(user, channel)`, а не просто `channel`.
- Лок `_lock_for(channel)` → `_lock_for(profile, channel)`.

MVP-сейчас: один аккаунт, один `org_id` из `.env` — этого достаточно.

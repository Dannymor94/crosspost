# ROADMAP.md — путь к MVP

**Цель MVP:** локальный инструмент с UI, где клиент подключает свои каналы (свои данные) 
и публикует контент в несколько площадок одним действием. Реальные данные, изоляция профилей.

**Что УЖЕ есть:** ядро работает — 4 канала (Telegram, Яндекс, ВК-стена, ВК-канал) постят из CLI. 
Фундамент данных готов (Слой 0: SQLite, vault, профили с изоляцией, валидация подключений).

**Принцип:** идём крупными итерациями. Каждая = законченный вертикальный срез, который "работает", 
а не "написан код". Не мельчим.

---

## Итерация 1 — UI подключения (Слой 1)

**Даёт:** клиент через интерфейс подключает каналы своими данными, видит живые статусы.

- Экран профилей (клиенты, переключение, создание).
- Карточки каналов со статус-индикатором (live / needs_relogin / не подключён).
- Подключение: API-канал → форма с полями + инструкции "где взять"; браузерный → кнопка "Войти".
- "Проверить и сохранить" → реальная валидация → статус по факту.
- Стек: React + FastAPI, дизайн по frontend-design (приборная панель оператора).

**Готово, когда:** можно завести профиль, подключить все 4 канала через UI (не CLI), 
статусы отражают факт.

---

## Итерация 2 — Публикация через UI (Слой 2, ядро MVP)

**Даёт:** главный экран — загрузил контент, выбрал каналы, отправил, видишь результат.

- Экран публикации: контент (текст + фото/видео) + выбор каналов (только подключённые, 
  живые, поддерживающие тип) + "Опубликовать".
- Асинхронные поканальные статусы: загораются по одному (✓/✗ по каналу).
- Частичный успех — норма; кнопка "Повторить" у упавшего канала.
- Публикация идёт через готовые адаптеры под нужным профилем.

**Готово, когда:** через UI один контент уходит в выбранные каналы, результат виден поканально. 
**Это функциональный MVP** — им уже можно пользоваться.

---

## Итерация 3 — Надёжность и диагностика (Слой 2, продолжение)

**Даёт:** инструмент, который можно отдать реальным людям и обслуживать.

- Диагностическое логирование для владельца: структурный лог каждой публикации 
  (профиль, канал, результат, где упало) + скриншот при ошибке браузерного канала.
- Клиенту — человеческие сообщения; владельцу — технические детали.
- Humanize: рандомные задержки, посимвольный ввод, паузы (снижение риска бана).

**Готово, когда:** при поломке у клиента владелец видит в логах что и где сломалось.

---

## Итерация 4 — Планировщик (отложенный постинг)

**Даёт:** посты по контент-плану, а не только "сейчас".

- Отложка: запланировать пост на время, хранит план.
- Пейсинг: в пиковой точке — поочерёдно с интервалом (устойчивость > скорость, анти-бан).
- Цепочка отложка → пейсинг → очередь → эфемерный браузерный воркер.

**Готово, когда:** запланированный пост уходит в заданное время; пики растягиваются поочерёдно.

## После MVP (по мере надобности)

- Новые каналы по рецепту: Telegraph, YouTube, WhatsApp, Instagram, Дзен (см. PROJECT_OVERVIEW §7).
- Типы контента: article, reel, story.
- Реальные external_id для vk_channel/yandex (сейчас заглушки).
- ВК-фото через API под user-токен (флаг готов).
- Перенос на VPS (headless/Xvfb, прокси на аккаунт).

---

## Границы MVP (что НЕ делаем до готовности ядра)

- Не тащим новые каналы, пока UI+публикация+планировщик не закрыты на текущих 4.
- Не строим коробочную версию (только локальная).
- Не делаем веб-аутентификацию клиентов в сервис (локальный мультипрофиль).
- Не гонимся за идеальной анти-бан эмуляцией (humanize = разумный слой, не гарантия).

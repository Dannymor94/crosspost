# MILESTONES.md — вехи и граница MVP

Цель документа: дать Claude Code **минимальный путь до работающего поста** и явный стоп,
чтобы не строить инфраструктуру раньше валидации ядра.

Принцип: каждая веха — это «оно работает», а не «написан код». Не уходить вперёд, пока текущая не зелёная.

---

## MVP-0 — Walking skeleton (САМОЕ ВАЖНОЕ, делать первым)

**Что:** CLI постит один контент в Telegram и ВК.

```
python -m crosspost post --type post --text "..." --image path.jpg --to telegram,vk
```

**В составе:**
- 2 реальных API-адаптера: Telegram (Telethon), ВК (VK API).
- Синхронный вызов адаптеров по списку каналов.
- Состояние и идемпотентность — **JSON-файл** (`JSONIdempotencyStore`). Не БД.
- Capability-проверка перед отправкой.
- Поканальный вывод в консоль: `✓ telegram (id=…) · ✗ vk (ошибка…)`.

**ЧЕГО НЕТ (жёсткий стоп для MVP-0):**
- ❌ Очередь (Redis/Celery) — синхронно.
- ❌ Веб/FastAPI — только CLI.
- ❌ БД (SQLite/Postgres) — только JSON.
- ❌ Браузерный тир (Playwright) — ни одного браузерного канала.
- ❌ vault, breaker, планировщик, мультиюзер.

**Готово, когда:** `make smoke` с реальными ключами постит в оба канала и печатает квитанции.

## MVP-0.5 — Остальной API-тир

Добавить Telegraph (`article`) и YouTube (`reel`) по «рецепту адаптера» (см. PLAN.md).
Всё ещё CLI + JSON, без инфраструктуры. Граница API↔браузер не пересекается.

**Готово, когда:** CLI постит во все 4 API-канала, каждый адаптер зелёный на моках.

## MVP-1 — Рабочий инструмент (инфраструктура как улучшение)

Теперь — и только теперь — инфраструктура оправдана:
- Очередь + раздельные fast/slow дорожки (эпик 3), переход JSON → Postgres.
- Веб-интерфейс: форма + поканальные статусы + ретрай (эпик 4).
- Медиа-цикл с TTL (эпик 7).

**Готово, когда:** через веб-форму уходит мультиканальная отправка с асинхронными статусами и точечным ретраем.

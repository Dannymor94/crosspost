# PROJECT_OVERVIEW.md — Кросс-постинг

**Версия:** 2.0
**Статус:** 4 канала работают вживую; переход от CLI к UI-сервису.

---

## 1. Что это

Инструмент кросс-постинга: один готовый контент (текст + фото/видео) публикуется в несколько площадок одним действием. Контент система **доставляет, а не генерирует**.

Использование: локально, мультипрофиль — несколько реальных клиентов, у каждого свои аккаунты и подключения, изолированно друг от друга. Обслуживание («под капотом») — на стороне владельца.

---

## 2. Как мы к этому пришли

Проект прошёл путь от идеи до рабочего инструмента, и ключевые решения родились из практики, а не из теории:

- **Начали с постинга текст+фото** в TG, ВК, Яндекс. Быстро выяснилось: у каналов разная природа доступа.
- **ВК через API упёрся во все стены** — токен сообщества не грузит фото и не постит текст (ошибки 27, 214), user-токен через официальный поток недоступен без плясок. Вывод: часть каналов API не отдаёт.
- **Решение: граница API↔браузер.** Где есть рабочий API — идём через него (надёжно). Где нет — автоматизируем живой браузер (хрупко, но работает везде).
- **Браузерный тир доказан вживую** на трёх непохожих потоках (Яндекс-форма, ВК-стена-модалка, ВК-канал-composer) — значит инфраструктура универсальна.
- **Сессии — через storageState**, не persistent-профиль (Яндекс/ВК не «прилипали» к профилю). Релогин — ручной, пароли не храним.
- Из этого выросли принципы детерминизма, диагностики и продуктовая модель ниже.

---

## 3. Главный принцип: граница API ↔ браузер

Единственная ось, объясняющая всё — надёжность, релогин, IP, стоимость, способ подключения.

**API-тир** (надёжный, дешёвый, stateless): токены/ключи, публикация возвращает квитанцию. Подключение = ввод полей.

**Браузерный тир** (хрупкий, дорогой): живой Playwright-браузер, сессия через storageState, нет квитанции. Подключение = вход через окно браузера.

Стратегия: **вытаскивать каналы из браузера в API везде, где возможно** — каждый переехавший канал исчезает из списка проблем.

## 5. Архитектура: детерминированное ядро + декларации каналов

Ключ к надёжности и к «чиню под капотом я» — максимальная детерминация:

- **Канал = декларация, не код.** Тир, поля ввода, инструкции, селекторы, способ валидации — фиксированная структура. Добавить/починить канал = поменять декларацию. Логика одна на всех.
- **Один путь исполнения:** `валидация → рендер → сессия → действие → verify → статус`. Без ветвлений «если ВК — так, если Яндекс — эдак». Различия каналов — в декларациях, конвейер один.
- **Явные состояния (конечный автомат).** Задача: `new → queued → attempting → done | failed | needs_relogin`. Подключение: `live | needs_relogin | banned`. Переходы жёстко заданы.
- **Хрупкое — в одном месте на канал.** URL, селекторы, тексты кнопок — константами наверху файла канала.
- **Идемпотентность по внутреннему ключу.** `publication_id` до отправки, дедуп по `(id, канал)`. Повтор не создаёт дубль.
- **Никакой «сообразительности» в рантайме.** Не нашёл селектор → чёткая ошибка + лог + скриншот, а не «попробуем иначе». Предсказуемый провал лучше непредсказуемого успеха.

Итог: **вся вариативность — в декларациях каналов, вся логика — единый детерминированный конвейер.**

## 6. Продуктовая модель

**Режим один — локальная мультипрофильная система.** Коробочной версии нет. Одно ядро; клиенты — изолированные наборы данных внутри одного приложения.

**Два уровня сложности (принцип):**
- **Клиент — без танцев с бубном.** Видит поля, вставляет цифры/ссылки по инструкции, жмёт «Подключить». Для браузерных каналов — кнопка входа, окно браузера, залогинился. Никаких терминалов.
- **Под капотом — только владелец.** Селектор поехал, площадка сменила форму — чинит владелец в коде, выкатывает обновление. Клиент не касается. Хрупкость браузерного тира лежит на владельце, где её можно лечить (главный довод за хостинг, а не коробку).

**Реальные данные с самого начала:**
- шифрованное хранилище учёток/сессий (vault, ключ снаружи) — сразу, не «потом»;
- проверка подключения при вводе (ввёл токен → тестовый запрос → зелёный/красный по факту);
- изоляция профилей — данные клиента A недоступны из контекста B;
- статусы отражают фактическое состояние сессии, а не «когда-то подключали».

**Надёжность:**
- **Humanize** — рандомные задержки между шагами, посимвольный ввод, паузы перед кликами. Снижает риск бана, но не гарантия; реальная защита — редкость + разные сессии + (позже) прокси на аккаунт.
- **Диагностическое логирование для владельца** — структурный лог каждой публикации (клиент, канал, результат, где упало) + скриншот при ошибке браузерного канала. Собирается к владельцу; клиент видит только человеческие сообщения. Это «пульт диагностики», без него хостовая модель не работает.

**Хранилище — SQLite** (конкурентная запись параллельных профилей, связанные таблицы профиль→каналы→статусы→сессии→логи). Шифрование учёток — отдельным слоем поверх.

**Слои реализации:**
- **Слой 0 — фундамент:** SQLite, модель профилей + изоляция, vault, ChannelConnection, валидация подключения.
- **Слой 1 — UI подключения:** профили (переключение), карточки каналов со статусом, формы API (поля + инструкции), кнопка-вход для браузерных.
- **Слой 2 — публикация + диагностика:** экран публикации, поканальные статусы, логирование, humanize.

**Мощности (ориентир):** API-тир почти бесплатен. Браузерный — эфемерный Chromium ~300–700 МБ на публикацию. При редком постинге на 3–5 клиентов достаточно VPS 4 ГБ / 2 vCPU. На сервере — headless или headful через Xvfb; при масштабе — прокси на аккаунт.

## 7. Будущие каналы и как реализуем

Логика подключения везде одинаковая — меняется только тир и декларация. ВК это ВК, Яндекс это Яндекс.

| Канал | Тир | Контент | Подход к реализации |
|---|---|---|---|
| **Telegraph** | API | статья | Токен одним GET-запросом. Публикация статьи через API. Расширяет на тип `article`. |
| **YouTube** | API | видео/Shorts | Data API + OAuth. Заливка видео. Учесть суточную квоту. Тип `reel`. |
| **WhatsApp** | браузер | Status / группа | По рецепту браузерного адаптера. Нет понятия «пост» — Status или сообщение в группу. Самый баноопасный — humanize обязателен. |
| **Instagram** | браузер | Reels / Stories | Браузер (личный аккаунт). «Добавить рилс». Хрупкий UI (обрезка, обложка). Либо Graph API при бизнес-аккаунте (тогда переезд в API-тир). |
| **Дзен** | браузер | статья | API на постинг нет. Редактор статьи через браузер, либо мост из TG-канала. |
| **MAX** | API | сообщение | Bot API есть, но публикация ботов — только верифицированным юрлицам РФ. Без юрлица — отложено. |
| **ВК — фото через API** | API | — | Требует пользовательский токен (wall+photos). Код готов за флагом `VK_PHOTO_UPLOAD_ENABLED`. Снять флаг при появлении токена. |

**Рецепт добавления браузерного канала** (проверен на Яндексе и ВК):
1. Пройти путь публикации руками, собрать DOM ключевых элементов (форма, поле, кнопка отправки, превью).
2. Вынести селекторы в константы (приоритет `data-testid` / текст / role, не генерённые классы).
3. Написать адаптер по образцу (тот же контракт, storageState-сессия, verify по факту публикации).
4. Ручной логин через `login`, живой прогон, отладка селекторов под реальный DOM.
5. Прописать в capability-матрицу и декларацию канала.

**Рецепт добавления API-канала:**
1. Получить способ аутентификации (токен/OAuth), описать поля и инструкции «где взять».
2. Написать адаптер (контракт, идемпотентность по внутреннему ключу, ленивый импорт SDK).
3. Валидация подключения при вводе. Прописать в capability-матрицу.

**Типы контента (расширение):** `post` (текст+фото) → `article` (лонгрид) → `reel` (постоянное видео) → `story` (эфемерное 24ч). Каждый рендерится под канал отдельно; дублировать один контент во все каналы нельзя — площадки режут охват за копипасту.

## 8. Стек

Python, FastAPI (UI-бэкенд), SQLite, Playwright (браузерный тир), Telethon (TG), шифрованный vault. Всё async. Детерминированный конвейер, декларативные каналы.

## 9. Модель ресурсов и планировщик

**Два процесса с разной жизнью:**
- **UI-сервер** (FastAPI) — лёгкий, живёт постоянно. Гасить нет смысла: экономия копеечная, а клиент должен иметь доступ к настройкам/статусам в любой момент.
- **Браузерный воркер** (Chromium) — тяжёлый, **эфемерный by design**: поднимается на публикацию, гасится сразу. В простое его в памяти нет. Здесь вся экономия, и она уже заложена.

**Машину (VPS) под расписание не гасим:** облачная оплата за аренду, а не за CPU-секунды; старт с нуля — минуты и хрупко; клиент не сможет зайти в «спящее» время. Экономия иллюзорная, риск недоступности перевешивает.

**Планировщик = отложка → пейсинг (цепочка, не два независимых скрипта):**
- **Отложка** — знает контент-план (известен заранее), держит запланированные задачи, отдаёт готовые в нужный момент.
- **Пейсинг (распределитель нагрузки)** — в пиковой точке, когда сходится много постов, **не пускает залпом**, а выпускает **поочерёдно** с интервалом.

**Поочерёдно, а не параллельно — приоритет «устойчивость > скорость»:**
- устойчивость: 1–2 браузера за раз не роняют память; залп из многих — риск падения;
- анти-бан: одновременные действия с фермы = сигнал, поочерёдно с паузами = норма;
- скорость не критична: задержка в минуты допустима, пик спокойно растягивается во времени.

Контент-план известен → нагрузку можно считать наперёд: пейсинг заранее знает пики и детерминированно растягивает их. Цепочка: `отложка → пейсинг → очередь → воркер`, каждый браузер гасится после себя.

**Проект:** Кросс-постинг с поканальной адаптацией контента
**Версия:** 1.1
**Статус:** дизайн зафиксирован, кодинг не начат

> Изменения v1.1: исправлена модель идемпотентности (внутренний ключ, не `external_id`); разведены состояния подключения и задачи; добавлены capability-матрица и circuit breaker; уточнены ключ vault, выбор БД и асинхронность контракта.

---

## 1. Что это

Личный инструмент кросс-постинга для SMM. Один экран: загружаешь контент, отмечаешь каналы, жмёшь «Отправить» — система раскладывает публикацию по выбранным площадкам и показывает поканальный результат.

Сначала для себя, затем для 3–5 друзей. Локальное / самохостируемое решение, ориентированное на работающие в РФ каналы, без зависимости от заблокированных облачных API.

**Не** автономный SMM-агент и **не** генератор контента. Контент готовый, на входе. Задача системы — доставка, а не творчество.

---

## 2. Контент

Два типа на входе:

- **Пост** — текст + картинка
- **Рилс / сторис** — текст + видео

Канонический объект контента несёт поле `type`: `post` / `article` / `reel` / `story`. Это разные модели данных (у статьи — заголовок, тело с разметкой, обложка; у поста — подпись + изображение; у рилса — постоянное вертикальное видео; у сторис — эфемерное на 24 ч), их нельзя сливать в один тип.

---

## 3. Главный принцип: граница API ↔ браузер

Каналы делятся не по «соцсетям», а по одной оси: публикуется через **API** или через **браузер**. Эта граница объясняет всё остальное — релогин, IP, прокси, надёжность, стоимость масштабирования, мультиюзерность. Любое решение сверяется с ней.

### API-тир — надёжный, дешёвый, stateless

| Канал | Способ |
|---|---|
| Telegram | Telethon (userbot), постинг от лица канала |
| ВКонтакте | VK API |
| YouTube | Data API (видео) |
| Telegraph | API (статьи) |

Свойства: сессии практически бессмертны, релогин почти никогда, общий IP сервера допустим, публикация возвращает квитанцию `external_id`, горизонтальное масштабирование тривиально.

> Telegram — намеренно userbot (Telethon), а не Bot API: бот ходит на единственный `api.telegram.org`, который душится в РФ, тогда как MTProto использует гибкий транспорт (вплоть до MTProto-прокси). Цена: автоматизация личного аккаунта формально против ToS — для редких постов риск низкий.

### Браузерный тир — хрупкий, дорогой, требует ухода

| Канал | Особенность |
|---|---|
| WhatsApp | вкладка; нет понятия «пост» — Status / группы; высокий риск бана |
| Instagram | вкладка, «добавить рилс»; самый ломкий узел |
| Дзен | вкладка, редактор статьи; официального API на постинг нет |
| Яндекс Бизнес | вкладка, «Публикации»; API нет, модерация до 7 дней |

Свойства: сессии смертны, нужен пользовательский релогин, липкий прокси на аккаунт, шифрованный vault, **квитанции нет** (проверка факта публикации только чтением ленты).

**Раскладка финальная: 4 канала через API, 4 через браузер. Граница ровно посередине.**

### Стратегический вектор

Вытаскивать каналы из браузера в API везде, где возможно (напр. мост TG→Дзен, бизнес-аккаунт IG ради Graph API). Каждый канал, переехавший влево, исчезает из списка проблем разом.

## 4. Контент-слой

**Canonical Content** (одна сущность с полем `type`) → рендер под каждую площадку.

Конвейер один и тот же (валидация → рендер → публикация → статус), но на выходе каждого канала контент свой. Дублировать один и тот же текст везде нельзя: площадки режут охват за копипасту. Пост для Telegram провалится в Дзене, где нужен лонгрид с подзаголовками; текст для Дзена слишком длинный для ВК.

**Capability-матрица.** Не каждый канал принимает каждый тип: YouTube не берёт `post`, Telegraph не берёт `story`, и т.д. Матрица `канал → множество поддерживаемых type` — несущий артефакт: ею определяется, какие каналы вообще доступны для данного контента (какие чекбоксы активны), и она валидирует отправку **до** очереди. Без неё UI предложит бессмысленные комбинации, которые уедут в очередь как ошибки.

**Рендер физически живёт в файле адаптера** (см. структуру) до тех пор, пока логика рендера не начнёт делиться между каналами — тогда выделяется отдельный `Renderer`. Преждевременный сплит «рендерер + адаптер» на каждый канал не делаем.

---

## 5. Поток пользователя

1. Загрузил контент (текст + картинка, либо текст + видео).
2. Доступны только каналы, поддерживающие этот `type` (по capability-матрице) и с живым подключением.
3. Отметил каналы чекбоксами.
4. Жмёт «Отправить».
5. Задачи падают в очередь; экран сразу показывает «отправляется…» поканально.
6. Статусы загораются по одному по мере готовности: `✓ ВК · ✓ TG · ✗ Instagram (нужен релогин)`.
7. У упавшего канала — кнопка «Повторить только его» (точечный дослал, без дублей в успешные).

Плюс отложенный постинг по расписанию.

Поведение под капотом: **асинхронно, поканально, с предварительным статусом канала и точечным ретраем.** Отправка не мгновенная (браузерный тир — десятки секунд + ретраи), синхронного ожидания всех каналов нет.

---

## 6. Инфраструктура

**Оркестратор + очередь.** Принимает отправку, раскладывает на поканальные атомарные задачи, ретраит (с потолком попыток), собирает статусы. Раздельные fast / slow дорожки — чтобы зависший браузерный канал не держал мгновенный API-канал. Перед диспатчем сверяется состояние подключения и circuit breaker.

**Состояние — две сущности.** `ChannelConnection (user, channel)`: `live | needs_relogin | banned` — состояние сессии/токена, хранится один раз на подключение. `Task` — состояние одной публикации в канал: `new → queued → attempting → done | failed(reason) | held`. Идемпотентность — по внутреннему `publication_id`, сгенерированному до отправки; `external_id` — лишь квитанция об успехе.

**Circuit breaker.** Сигнал бана/жёсткого лимита от площадки → подключение в `banned`/пауза, задачи в него не диспатчатся. Защита от того, чтобы ретраями углублять бан. Прямое следствие самого смысла проекта.

**Планировщик** — цепочка: `отложка` (когда задача готова) → `пейсинг` (с каким интервалом впускать, чтобы не залпом) → очередь. Пейсинг — разнос нагрузки + анти-бот-мера для фермы. Связь явная, не два независимых скрипта.

**Браузерная ферма.** Изолированные персистентные профили (`user-data-dir`), **не инкогнито** (инкогнито = разлогин перед каждым постом). Воркер поднимает профиль по требованию, постит, гасит. Четыре браузерных канала = четыре независимых адаптера с общим интерфейсом `publish(canonical) → result`. Открытый вопрос: headless vs headful через Xvfb на сервере.

**Медиа.** Живёт локально (temp) до отправки. Стирается только когда **все** выбранные каналы финально завершились — не по первому успеху. Дополнительно TTL, чтобы файл не пинился бесконечно из-за висящего `needs_relogin`-канала. Дальше «хранится» самими площадками; постоянного блоб-хранилища нет.
> IG идёт через браузер (file picker берёт локальный файл) → временная публичная ссылка для IG-видео не нужна. Если IG переедет на Graph API — потребуется temp-хостинг под видео.

**Vault.** Шифрование сессий at rest. **Ключ шифрования инжектится снаружи (env/keyring/KMS) и НЕ лежит под `runtime/` рядом с зашифрованными данными** — иначе копия `runtime/` уносит и замок, и ключ. Расшифрованная сессия живёт в памяти воркера коротко. Доступ — только воркерам, не фронту. SPOF и главная цель атаки.

## 7. Мультиюзерность (3–5 друзей)

Граница API↔браузер определяет и её:

- **API-каналы** — масштабируются тривиально: токен / StringSession на пользователя, просто строки.
- **Браузерные каналы** — профиль + липкий прокси + релогин на каждого пользователя. Это и есть то, что превращает скрипт в сервис. При 3–5 друзьях подъёмно.

Один пользователь = один изолированный браузерный профиль (его соцсети — вкладки внутри). «Открытый браузер» на сервере читается как «залогиненный профиль на диске», которым управляет автоматизация, а не руки.

IP: общий допустим для API-тира; для браузерного — липкий per-account egress (резидентный/мобильный прокси), сессия всегда выходит с одного IP. Общий VPN на всех — нельзя (бан одного IP уносит всех).

## 8. Жёсткие правила

Выведены по ходу проектирования, нарушать нельзя:

1. **Идемпотентность по внутреннему ключу.** `publication_id` до отправки; источник истины — `(publication_id, channel)`. `external_id` — квитанция, не ключ дедупа.
2. **Verify-before-retry на браузерном тире.** Окно «ушло → упал до записи done» есть на обоих тирах; в браузере (нет квитанции) перед повтором читаем ленту канала. Слепой ретрай запрещён.
3. **Circuit breaker на бан/лимит.** Забаненный канал на паузе, не ретраится.
4. **Сырые пароли не храним.** Релогин user-driven (QR/код). Автологин паролем не реализуем.
5. **Лок `(user, channel)`** — одна задача на сессию.
6. **Чистка медиа — по финалу всех каналов** + TTL.
7. **Контент рендерится под канал и только в поддерживаемые типы** (capability-матрица).
8. **Проактивный health-check.** Мёртвое подключение → `needs_relogin`, задачи не ставятся.
9. **Ретраи с потолком попыток.**

Правила 2/3/5/8 — свойства браузерного тира как класса, применяются ко всем четырём браузерным каналам одинаково.

## 9. Стек (предполагаемый)

- Бэкенд — Python, FastAPI, **всё async**
- Очередь — Redis + Celery / RQ
- Браузер — Playwright (персистентные профили)
- Telegram — Telethon
- Состояние — Postgres (как только воркеров больше одного; SQLite только для одиночной фазы A)
- Веб-интерфейс — форма + поканальные статусы + кнопки «Повторить»
- Vault — шифрованное хранилище сессий, ключ инжектится снаружи

---

## 10. Точка старта

**TG-адаптер на Telethon** — первый канал API-ядра. Не зависит ни от очереди, ни от фермы, ни от vault, ни от планировщика. Интерфейс `async publish(canonical) → result`, на вход canonical content, на выход «опубликовано в канал» + квитанция. Даёт первый рабочий результат и задаёт контракт адаптера для всех остальных каналов.

## Что это

Локальный кросс-постинг: загрузил контент → выбрал каналы → опубликовалось → поканальный результат.
Контент **готовый, на входе**. Система доставляет, а не генерирует. Не добавлять генерацию контента.

## Главный инвариант: граница API ↔ браузер

Каждый канал — по одну сторону границы. Не смешивать.

- **API-тир** (Telegram/Telethon, ВК, YouTube, Telegraph): stateless, общий IP, площадка возвращает квитанцию (`external_id`).
- **Браузерный тир** (WhatsApp, Instagram, Дзен, Яндекс): персистентные профили, лок сессии, прокси на аккаунт, квитанции нет.

Никогда: браузерной логики в API-адаптере; прокси/сессий у API-адаптера; общего скрипта на несколько браузерных каналов.

## Контракт адаптера

Каждый из 8 каналов реализует один интерфейс: **`async def publish(canonical) -> ChannelResult`**.
Контракт асинхронный с самого начала (Telethon и Playwright асинхронны) — не делать синхронным «на время».

- Новый канал = новая реализация контракта. **Не** трогать оркестратор, **не** спецкейсить канал в ядре.
- Адаптеры изолированы: падение/редизайн одного не должен ломать другие.
- `ChannelResult = {status, external_id?, error?, channel}`. `external_id` — **квитанция об успехе** (для верификации/правки/удаления), НЕ ключ дедупа.

---

## Две сущности состояния — не путать

Релогин и бан — свойства **подключения к каналу**, а не отдельной публикации.

- **ChannelConnection** `(user, channel)`: `live | needs_relogin | banned`. Состояние сессии/токена.
- **Task** (публикация в канал): `new → queued → attempting → done | failed(reason) | held`.

Задача упирается в мёртвый канал и встаёт `held` или падает `failed(reason=needs_relogin)`. Само «нужен релогин» хранится **один раз на подключении**, а не на каждой задаче.

## Жёсткие правила (нарушать нельзя)

1. **Идемпотентность через внутренний ключ.** `publication_id` генерится **до** отправки. Источник истины — состояние `(publication_id, channel)`. Перед диспатчем: канал уже `done`? — пропустить. `external_id` приходит **после** успеха и дедуп не обеспечивает.
2. **Окно «ушло → упал до записи done» есть на ОБОИХ тирах.** Закрываем по-разному: API-тир — внутренний ключ + идемпотентный ключ площадки, где он есть; браузерный тир — **verify-before-retry** (маркер `attempting` до клика, при повторе сперва прочитать ленту канала, потом дожимать). Слепой ретрай в браузере запрещён.
3. **Бан/жёсткий лимит → circuit breaker.** Поймали сигнал бана/лимита на `(user, channel)` → подключение в `banned`/пауза, задачи **не диспатчатся**. Ретрай в забаненный канал запрещён — он углубляет бан.
4. **Сырые пароли не хранить нигде.** Релогин — только user-driven (QR/код пользователем). Авто-логин паролем не реализовывать.
5. **Лок `(user, channel)`** перед любым доступом к сессии. Одна задача на сессию в момент времени.
6. **Чистка медиа** — только когда ВСЕ выбранные каналы финально завершились (done/failed после ретраев). Не по первому успеху. Плюс TTL: медиа не пинится бесконечно из-за висящего `needs_relogin`-канала.
7. **Контент рендерится под канал** и **только в поддерживаемые типы.** Сверяться с capability-матрицей (канал → множество типов). Не отправлять один рендер во все каналы; не диспатчить `reel` в канал, который его не принимает.
8. **Частичный успех — норма.** Статусы поканальные, асинхронные. Ретрай — только упавшего канала, без повторной отправки в успешные. У ретраев есть потолок попыток.
9. **Здоровье сессий проактивно.** Health-check по крону до отправки. Мёртвое подключение → `needs_relogin`, задачи в него не ставятся в очередь (явная деградация, не тихая потеря поста).

## Стек и конвенции

- Backend: Python, FastAPI. Очередь: Redis + Celery/RQ. Браузер: Playwright. TG: Telethon. Всё async.
- Состояние: БД выбирается по **конкурентности записи**, не по числу юзеров. Несколько воркеров очереди пишут статусы конкурентно → SQLite упрётся в write-lock уже на MVP-1. Postgres — как только воркеров больше одного.
- Секреты — в `runtime/.env`, не коммитить. **Ключ шифрования vault инжектится снаружи (env/keyring/KMS), НЕ лежит файлом рядом с vault.**
- Очереди раздельные: fast (API) / slow (браузер) не блокируют друг друга.

## Как работать

- **TDD на адаптерах.** Дубли и баны стоят реальных денег и аккаунтов друзей — сперва тест (мок площадки → идемпотентность по внутреннему ключу, verify-before-retry, реакция на бан), потом код.
- Минимализм. Не усложнять ядро ради канала. Рендер и публикация канала живут в **одном файле адаптера**, пока логика рендера не начнёт делиться между каналами — тогда выделять `Renderer`. Не плодить слои заранее.
- Детерминизм оркестрации: поведение явное. Адаптеры — тупые и изолированные.
- При сомнении — сверься с границей API↔браузер. Она объясняет почти любое решение здесь.

## Не делать

- Не сливать адаптеры в общий скрипт.
- Не класть браузерную инфраструктуру (сессии/прокси/релогин) в API-тир.
- Не хранить пароли, не логиниться паролем автоматически.
- Не ретраить браузерный пост вслепую; не ретраить забаненный канал.
- Не использовать `external_id` как ключ дедупа.
- Не держать ключ vault под `runtime/`.
- Не дублировать сюда содержимое PROJECT_GUIDE/PRD_BACKLOG. Этот файл держать коротким.
